Propósito del documento El presente documento tiene como propósito especificar los requisitos funcionales y no funcionales del sistema Arriendos360, una plataforma web para la gestión de arrendamientos inmobiliarios. Este SRS establece el acuerdo formal entre el equipo de desarrollo y el cliente académico (profesor evaluador) sobre qué debe hacer el sistema. Sirve como base para el diseño, implementación y validación del producto, y como referencia para verificar que el software entregado cumple con lo especificado. Tabla SEQ Tabla \* ARABIC 1 . Este SRS cubre la especificación del Producto Mínimo Viable (MVP) de Arriendos360, limitado a las funcionalidades priorizadas como Must Have y Should Have según la técnica MoSCoW definida en el PMP (Sección 7.1.1). Alcance del producto Nombre del producto: Arriendos360 Descripción: Arriendos360 es una plataforma web diseñada para digitalizar la gestión de arrendamientos inmobiliarios, dirigida a pequeños y medianos propietarios en Colombia que actualmente administran sus inmuebles de forma manual o mediante hojas de cálculo. Problema que resuelve: Los propietarios de inmuebles en arriendo enfrentan dificultades para llevar un control organizado de contratos, pagos y vencimientos. Las alternativas actuales son: (1) gestión manual con alto riesgo de errores y olvidos, o (2) contratar inmobiliarias tradicionales con comisiones elevadas (8-10% del canon mensual). Arriendos360 ofrece una tercera opción: una herramienta digital accesible que automatiza las tareas administrativas sin costos de intermediación. Objetivos del producto: Reducir el tiempo dedicado a tareas administrativas de arrendamiento en al menos un 50%. Centralizar la información de contratos y pagos en una única plataforma accesible desde cualquier dispositivo. Proveer visibilidad en tiempo real del estado financiero del negocio de arrendamiento. Beneficios esperados: Para propietarios: Control organizado, alertas automáticas, reducción de morosidad. Para inquilinos: Transparencia en su historial de pagos y acceso a información de su contrato. Exclusiones (Won't Have): El MVP no incluye: facturación electrónica DIAN, pasarela de pagos en línea, notificaciones por SMS, ni integraciones con entidades bancarias. Estas funcionalidades se consideran para versiones futuras. Relación con objetivos académicos: Este proyecto se desarrolla como parte del curso Ingeniería de Software I, aplicando metodologías de gestión de proyectos, desarrollo iterativo incremental y buenas prácticas de ingeniería de requisitos. Perspectiva de producto El sistema Arriendos360 es un producto de software totalmente nuevo, concebido como una plataforma web independiente. Surge para brindar una alternativa digital a los pequeños y medianos propietarios en Colombia que actualmente gestionan sus inmuebles de forma manual, en hojas de cálculo, o a través de inmobiliarias tradicionales que cobran comisiones elevadas. El desarrollo de este producto se justifica por la necesidad imperante de automatizar tareas administrativas, buscando reducir el tiempo de gestión dedicado a los arrendamientos en al menos un 50%. Además, permite centralizar la información de contratos y pagos en una única plataforma accesible desde cualquier dispositivo, sin incurrir en costos de intermediación. Los beneficios directos para los usuarios finales (arrendadores e inquilinos) incluyen un control organizado, alertas automáticas de vencimiento, reducción de morosidad y total transparencia en el historial de pagos. Interfaces con el sistema Dado que Arriendos360 es un sistema nuevo e independiente en su fase de Producto Mínimo Viable o MVP, actualmente no expone interfaces públicas ni APIs para la comunicación directa con sistemas externos o de terceros (como entidades bancarias o los sistemas de facturación electrónica de la DIAN, los cuales están excluidos en esta versión). La interacción principal a nivel de arquitectura de sistema se da de manera interna: el Frontend, desarrollado como una Single Page Application (SPA) en React (Meta Platforms, I., 2026), se comunicará de manera exclusiva con el Backend, construido en Node.js con Express. Esta comunicación se realizará mediante el consumo de una API REST propia. Interfaces con el usuario La interfaz de usuario de Arriendos360 será de tipo web, accesible a través de navegadores modernos como Google Chrome, Mozilla Firefox y Microsoft Edge. Se prioriza el diseño visual y la experiencia de usuario para entornos de escritorio, asegurando al mismo tiempo una funcionalidad básica adaptativa para el acceso mediante dispositivos móviles. La interacción se realizará mediante teclado, ratón o pantalla táctil, y la plataforma presentará interfaces y vistas diferenciadas según el nivel de seguridad y rol del usuario autenticado: Interfaz del Propietario (Arrendador): Dará acceso a un dashboard administrativo interactivo donde podrá visualizar estadísticas del negocio, gestionar contratos mediante operaciones CRUD, registrar pagos y recibir alertas del sistema. Interfaz del Inquilino (Arrendatario): Presentará un entorno simplificado y de solo lectura que permitirá al usuario consultar la información de su contrato vigente y visualizar su historial de pagos. Interfaces con el hardware Al ser una plataforma web, Arriendos360 no requiere integración directa ni comunicación a bajo nivel con hardware especializado por parte del cliente final. La interacción física se limita a los periféricos estándar de entrada y salida (pantallas, teclados, ratones, pantallas táctiles) que proporciona el dispositivo desde el cual el usuario ejecuta el navegador. Del lado del servidor, el software interactuará indirectamente con la infraestructura de hardware virtualizada proporcionada por el servicio de alojamiento de Microsoft Azure App Services. Interfaces con el software El sistema Arriendos360 se comunicará con otros productos de software esenciales para su ejecución, despliegue y, particularmente, para la persistencia de los datos. La interacción más crítica ocurre entre el Backend y el Sistema Gestor de Base de Datos (DBMS), la cual se gestionará mediante cadenas de conexión seguras a través de variables de entorno, utilizando controladores nativos o un ORM configurado en el entorno del servidor. Restricciones de memoria El despliegue y la operación del sistema están sujetos a restricciones estrictas de almacenamiento y memoria dictadas por el presupuesto académico del proyecto y el uso de recursos gratuitos: Almacenamiento en Base de Datos: La instancia de PostgreSQL operará bajo el "Free tier", lo que impone un límite de almacenamiento aproximado de 500 MB. Esto restringe el volumen total de registros históricos que pueden conservarse. Almacenamiento de Archivos: El sistema no cuenta con memoria de almacenamiento persistente para archivos físicos. Los documentos como contratos en PDF no se guardan en el servidor; en su lugar, se almacenan estrictamente como metadatos de texto en la base de datos. Memoria del Cliente: Se requiere que el dispositivo del usuario final disponga de suficiente memoria RAM libre para cargar la aplicación SPA en el navegador sin degradación del rendimiento del sistema operativo. Operaciones Modos de operación : El sistema discrimina sus operaciones basándose en la autenticación (mediante JWT) y el rol del usuario. El Propietario opera en "Modo Administrador" gestionando el ciclo de vida de los datos, mientras que el Inquilino opera en "Modo Usuario Final" con privilegios limitados de consulta. Periodos de actividad e inactividad: En condiciones normales, el sistema se espera que esté activo 24/7. Sin embargo, debido a las políticas del entorno de alojamiento gratuito de Azure, el servidor entrará automáticamente en un estado de inactividad ("sleep mode") después de 15 minutos sin recibir tráfico HTTP. Esto requerirá un tiempo adicional de procesamiento (arranque en frío) cuando un usuario intente acceder tras dicho periodo. Procesos de recuperación y respaldo: Al tener una tolerancia a fallos limitada por el entorno gratuito, no se cuenta con sistemas de replicación automática en tiempo real. Las operaciones de respaldo dependerán de las herramientas básicas de exportación manual de datos provistas por PostgreSQL y el control de versiones en GitHub para el código fuente. Funciones del producto El sistema Arriendos360 integra un conjunto de funciones diseñadas para digitalizar y optimizar la gestión de arrendamientos inmobiliarios. Estas funciones se organizan en módulos interdependientes que garantizan la cobertura de las necesidades principales de propietarios e inquilinos. A continuación, se presenta una descripción detallada de cada función, acompañada de su propósito y alcance: 1. Gestión de Contratos Operaciones CRUD: El sistema permitirá crear, consultar, editar y eliminar contratos de arrendamiento. Alertas de vencimiento: Se generarán notificaciones internas cuando un contrato esté próximo a finalizar, reduciendo el riesgo de olvidos. Historial de contratos: Se mantendrá un registro estructurado de contratos vigentes y pasados, accesible para el propietario. Metadatos en BD: Los contratos se almacenarán como registros textuales en la base de datos, sin archivos físicos, optimizando el uso del espacio limitado. 2. Gestión de Pagos Registro manual de pagos: Los propietarios podrán ingresar los pagos realizados por los inquilinos, indicando fecha, monto y estado. Historial financiero: El sistema generará un historial de pagos por contrato, permitiendo identificar morosidad y calcular ingresos acumulados. Detección de pagos pendientes: Se resaltarán los periodos no cancelados, apoyando la reducción de la morosidad. Indicadores de cumplimiento: Se podrán visualizar métricas como porcentaje de pagos realizados vs. pendientes. 3. Dashboard Administrativo Visualización de estadísticas: El propietario contará con un panel que muestre indicadores clave como ingresos mensuales, ocupación de inmuebles y contratos próximos a vencer. Gráficas dinámicas: Se incluirán representaciones visuales (barras, líneas, circulares) para facilitar la interpretación de datos. Acceso rápido a información crítica: El dashboard servirá como punto central para la toma de decisiones, reduciendo tiempos de búsqueda. 4. Autenticación y Roles Registro e inicio de sesión: El sistema permitirá la creación de cuentas diferenciadas para propietarios e inquilinos. Roles diferenciados: Propietario: Acceso completo a las funciones administrativas (contratos, pagos, dashboard). Inquilino: Acceso restringido a la consulta de su contrato vigente y su historial de pagos. Seguridad: Se implementará autenticación mediante JWT y cifrado de contraseñas con bcrypt, garantizando la protección de credenciales. 5. Funciones transversales Notificaciones internas: Alertas dentro del sistema para vencimientos de contratos y pagos pendientes. Acceso multiplataforma: El sistema será accesible desde navegadores modernos en escritorio y dispositivos móviles. Escalabilidad académica: Aunque limitado en el MVP, la arquitectura permitirá futuras extensiones como pasarelas de pago o facturación electrónica. Características del usuario El sistema Arriendos360 está diseñado para dos perfiles principales de usuario: Propietarios (Arrendadores) e Inquilinos (Arrendatarios) . Cada uno presenta necesidades, privilegios y niveles de interacción distintos, lo que exige una diferenciación clara en la definición de sus características. La identificación de estos perfiles es fundamental para garantizar que la plataforma cumpla con los objetivos de usabilidad, seguridad y eficiencia. Además, permite establecer criterios de diseño que aseguren que la experiencia de usuario sea coherente con las capacidades técnicas y expectativas de cada actor. Propietario (Arrendador) Rol: Administrador del sistema. Privilegios: Acceso completo al dashboard, operaciones CRUD sobre contratos y pagos, recepción de alertas automáticas. Nivel técnico: Básico–intermedio. Se espera que maneje herramientas digitales comunes como navegadores web y formularios en línea. Frecuencia de uso: Regular, con sesiones semanales o mensuales para registrar pagos, actualizar contratos y consultar estadísticas. Objetivo principal: Reducir el tiempo de gestión administrativa y centralizar la información de sus inmuebles. Inquilino (Arrendatario) Rol: Usuario final con acceso restringido. Privilegios: Consulta de su contrato vigente y visualización de su historial de pagos. Nivel técnico: Básico. No requiere conocimientos avanzados; basta con habilidades mínimas de navegación web. Frecuencia de uso: Ocasional, principalmente para verificar pagos realizados o consultar condiciones contractuales. Suposiciones y dependencias El desarrollo y operación del sistema Arriendos360 se fundamenta en un conjunto de supuestos y dependencias que condicionan su funcionamiento. Estas hipótesis permiten delimitar el alcance del proyecto y reconocer los factores externos que pueden afectar la estabilidad y evolución del sistema. Las suposiciones se refieren a condiciones que se consideran verdaderas en el contexto académico y técnico del proyecto, mientras que las dependencias identifican elementos externos de los cuales el sistema depende para garantizar su operatividad. Suposiciones principales Los usuarios (propietarios e inquilinos) cuentan con acceso estable a internet y utilizan navegadores modernos (Chrome, Firefox, Edge). Los propietarios ingresarán manualmente la información de contratos y pagos, dado que el MVP no contempla integración con APIs bancarias ni pasarelas de pago. El volumen de datos generado por los usuarios no superará el límite de almacenamiento de la base de datos PostgreSQL en modalidad gratuita (~500 MB). El sistema será utilizado principalmente en entornos académicos y de prueba, sin requerir escalabilidad empresarial inmediata. Se asume que los usuarios poseen un nivel básico de alfabetización digital suficiente para interactuar con formularios web y dashboards. Dependencias principales Dependencia del servicio gratuito de Microsoft Azure App Services, lo que implica limitaciones de disponibilidad (modo “sleep” tras 15 minutos de inactividad). Dependencia de la correcta configuración y mantenimiento de la base de datos PostgreSQL, que asegura la persistencia de contratos, pagos y usuarios. Dependencia del uso de GitHub para control de versiones y despliegues continuos, garantizando la trazabilidad del código fuente. Dependencia de librerías y frameworks externos (React, Node.js, Express, JWT, bcrypt), cuya estabilidad y soporte condicionan la seguridad y rendimiento del sistema. Dependencia de la participación de los tres integrantes del equipo, dado que el proyecto se desarrolla en un contexto académico con recursos humanos limitados. Tabla 13. Suposiciones y dependencias. Distribución de requisitos La distribución de requisitos del sistema Arriendos360 se organiza utilizando la técnica MoSCoW (Must Have, Should Have, Could Have, Won’t Have). Esta metodología permite priorizar las funcionalidades del Producto Mínimo Viable (MVP), asegurando que el desarrollo se enfoque en lo esencial dentro de las limitaciones de tiempo, recursos y alcance académico. La clasificación se realizó considerando : Impacto en los objetivos del proyecto: reducción del tiempo de gestión administrativa y centralización de la información. Valor para el usuario final: utilidad directa para propietarios e inquilinos. Restricciones técnicas y académicas: presupuesto nulo, hosting gratuito, tiempo limitado de 14 semanas. Tabla 15. Son esenciales para que el sistema cumpla su propósito principal de digitalizar la gestión de arrendamientos. Sin estas funciones, el producto carecería de utilidad práctica. Should Have (Deseables) - Alertas automáticas de vencimiento de contratos. - Historial detallado de pagos. - Indicadores de morosidad. Mejoran la experiencia del usuario y aportan valor agregado, pero no son críticos para la operación mínima del sistema. Could Have (Opcionales) - Exportación de reportes en PDF o Excel. - Personalización del dashboard. - Visualización avanzada de estadísticas con filtros dinámicos. Son funcionalidades que enriquecen el sistema, pero pueden posponerse para versiones futuras debido a las limitaciones de tiempo y recursos. Won’t Have (Excluidos en el MVP) - Facturación electrónica DIAN. - Pasarela de pagos en línea. - Notificaciones por SMS. - Integraciones con entidades bancarias. Se excluyen explícitamente en esta versión por su complejidad técnica, costos asociados y falta de relevancia inmediata en el contexto académico. Los Should Have se implementarán en la medida en que el tiempo y los recursos lo permitan, mientras que los Could Have y Won’t Have se reservan para futuras iteraciones o versiones comerciales del sistema. De esta manera, la distribución de requisitos establece un marco claro de trabajo, alineado con los objetivos académicos y las restricciones técnicas del proyecto. Trazabilidad: La trazabilidad busca que los requisitos se reflejen en las diferentes actividades del proyecto, permitiendo su verificación durante el desarrollo e implementación. Se distinguen dos tipos: Trazabilidad horizontal: Vincula cada requisito con las historias de usuario y otros requisitos relacionados. Por ejemplo, el requisito RF-01 (Registro de propietario) está asociado a la historia de usuario HU-01. Trazabilidad vertical: Vincula cada requisito con los artefactos de diseño, código fuente y casos de prueba. Por ejemplo, el requisito RF-01 se refleja en el mockup de la pantalla de registro, luego en el controlador AuthController.js, y finalmente en el caso de prueba Test_RF01. Proceso de verificación y validación El proceso de verificación y validación (V&amp;V) aplicado a la especificación de requisitos de Arriendos360 busca garantizar que cada requisito definido cumpla con los criterios de calidad establecidos y que el sistema final responda a las necesidades del usuario académico y funcional. La validación asegura que los requisitos expresan correctamente lo que el sistema debe hacer, mientras que la verificación confirma que dichos requisitos se implementan de manera consistente y comprobable durante el desarrollo. 10.1 Validación de requisitos Cada requisito identificado en el SRS se valida utilizando criterios de aceptación que permiten determinar su calidad y pertinencia. Los criterios aplicados son: Atómico: Cada requisito describe una sola funcionalidad o característica, evitando ambigüedad. Correcto: El requisito refleja fielmente una necesidad del sistema. No ambiguo: Está redactado de manera clara, sin interpretaciones múltiples. Completo: Incluye toda la información necesaria para su comprensión. Consistente: No entra en conflicto con otros requisitos definidos. Importancia: Se clasifica según su prioridad ( Must Have , Should Have , Could Have , Won’t Have ). Verificable: Puede ser comprobado mediante pruebas o inspecciones. Modificable: Está redactado de forma que pueda actualizarse sin afectar la coherencia del documento. Trazable: Se relaciona con historias de usuario y módulos específicos. Asociado a versión: Cada requisito se vincula a una versión del sistema. No redundante: Evita duplicidad con otros requisitos. Preciso: Define claramente el alcance y los límites de la funcionalidad. La validación se realiza mediante revisiones cruzadas entre los integrantes del equipo y el profesor evaluador, asegurando que cada requisito cumpla con estos criterios antes de ser aprobado. 10.2 Verificación de requisitos La verificación consiste en comprobar que los requisitos validados se implementan correctamente en el sistema. Para ello se aplican las siguientes actividades: Revisión documental: Contrastar cada requisito con el código y los entregables del sistema. Pruebas unitarias: Validar que cada requisito funcional (ej. CRUD de contratos, registro de pagos, dashboard ) se cumple de manera aislada. Pruebas de integración: Confirmar que los requisitos funcionan correctamente cuando los módulos interactúan entre sí (ej. Dashboard mostrando datos de contratos y pagos). Pruebas de aceptación: El profesor evaluador valida que los requisitos implementados cumplen con los objetivos académicos y funcionales. Verificación continua: Se realizan iteraciones semanales donde se revisa el cumplimiento de requisitos Must Have y Should Have , asegurando trazabilidad con las historias de usuario. 10.3 Diagramas de flujo del proceso Anexos Estimación del proyecto basado en Puntos de Función: Estimación del Proyecto.xlsx Referencias conceptosjuridicos.com.
